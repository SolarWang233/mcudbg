# 嵌入式革命开始：我让 AI 自己连接板子查 Bug

很多人都在说，AI 正在改变软件开发。

但如果你真的做嵌入式，就会很快发现一件事：
AI 最擅长的，还是“帮你写代码”；而嵌入式最难的，往往不是“把代码写出来”，而是“代码烧到板子上以后，到底发生了什么”。

比如一个看起来很简单的问题：

**板子灯不闪。**

这句话背后，可能藏着很多完全不同的原因：

- 板子根本没正常启动
- 初始化卡在某一步
- 程序已经跑飞
- 系统进了 HardFault
- 代码改了，但烧进去的并不是最新版本

也就是说，真正进入实战之后，你面对的已经不是“代码问题”，而是“现场问题”。

我最近在做的 `mcudbg`，就是想解决这类问题。

我想做的，不是让 AI 更会写一点嵌入式代码，而是让它开始真正参与“调板子”这件事。

## 这次我想验证的，不是概念，而是一条真板闭环

这次我录了一个真实 demo。

我没有让它做什么特别复杂的事情，只是给了它一句非常朴素的描述：

**板子灯不闪，帮我查一下。**

然后让它自己去做后面的事：

- 连接板子
- 读取串口日志
- 读取调试状态
- 判断问题发生在哪个阶段
- 定位故障原因
- 修改代码
- 重新编译
- 重新烧录
- 最后再验证问题是不是真的消失了

这次我最想证明的，不是“AI 能不能输出一段看起来像分析的话”，而是：

**它能不能在一块真板上，真的完成一次排障、修复、编译、烧录、再验证的闭环。**

如果这条链跑通了，那它就不再只是一个“会写代码的助手”，而是开始接近一个真正参与板级调试的工具。

## 表面现象是灯不闪，真正的问题却不是灯

这次的目标板是 `STM32L496VETx`。

我先让 `mcudbg` 去读板子的真实现场，而不是先盯着代码猜。

它走的链路很直接：

- 连接 `ST-Link`
- 连接 `UART`
- 加载 `ELF/AXF` 符号
- 复位板子
- 读取启动日志
- 结合 fault 寄存器和符号信息做诊断

结果很快就出来了。

表面现象是“灯不闪”，但真实问题并不是 LED 翻转逻辑写错了，而是程序在启动阶段就已经挂了。

这次板子实际打出来的关键日志是这样的：

```text
boot start
clock init ok
uart init ok
led init ok
sensor init...

[HardFault]
CFSR = 0x00000001
HFSR = 0x40000000
```

这几行日志其实已经说明了很多事：

- 板子已经启动起来了
- 时钟、串口、LED 初始化都走到了
- 程序跑到了 `sensor init...`
- 然后没有继续往下正常执行
- 系统进入了 `HardFault`

也就是说，灯不闪只是表象。

真正的问题是：

**系统在启动阶段就已经异常退出，后面的主循环和灯逻辑根本没机会稳定运行。**

## 它不是只会“看日志”，而是开始真正读板子

这次真正让我觉得事情开始有意思的，不是它看到了 `[HardFault]` 这几个字，而是它能把不同来源的现场信息拼起来。

它读到的不是单一信息，而是几层上下文：

- UART 启动日志
- probe 当前读到的寄存器状态
- fault 寄存器
- ELF 里的符号信息

最后它给出的结论不是一句模糊的“程序有异常”，而是更具体的诊断：

- `startup_failure_with_fault`
- `hardfault_detected`
- fault class: `instruction_access_violation`
- `PC` 已经落到 `HardFault_Handler`

这一步很重要。

因为一旦 AI 能把这些真实板级信息组织成一份有意义的结论，它就已经不再只是停留在“帮你读代码”这一层了。

它开始接触的是嵌入式里更麻烦、也更耗时间的那部分工作：

- 板子到底跑到哪一步了
- 是正常卡住，还是已经 fault 了
- fault 是启动路径问题，还是执行流跳飞了
- 当前看到的日志和寄存器状态是否互相印证

这也是我做 `mcudbg` 最核心的想法：

**不是做一个 probe 包装器，而是给 AI 一层板级调试和观测能力。**

## 只会指出问题还不够，关键是改完以后能不能证明它真的好了

如果这件事只停在“它能指出哪里出问题了”，那它依然只是一个不错的分析工具。

但对嵌入式来说，真正有价值的不是“分析一下”，而是：

**改完以后，再把板子跑起来，看它是不是真的恢复了。**

所以这次我没有停在故障诊断这里，而是继续让它往下走：

- 根据当前问题修改代码
- 重新编译
- 重新烧录
- 再复位一次
- 再读一遍启动日志
- 再判断系统是不是恢复正常

这一步对我来说，比前面的 fault 诊断还重要。

因为它决定了这件事到底是“会分析”，还是“开始具备闭环能力”。

## 修复后，我最想看到的不是一句成功提示，而是板子真的往下跑了

修复之后，我又让它重新走了一遍编译、烧录和验证流程。

这次板子打印出来的结果变成了：

```text
boot start
clock init ok
uart init ok
led init ok
sensor init...
sensor init ok
app loop running
```

对应的诊断结果也不再是 fault，而是：

`startup_completed_normally`

这意味着什么？

意味着这次不是“看起来像修好了”，而是从板子现场能确认：

- 启动路径继续往下执行了
- 没有再进入 `HardFault`
- 主程序已经进入正常运行状态

## 我觉得最有价值的，不是“AI 会写代码”，而是它开始接触真实系统

这件事让我更确定了一点：

AI 在嵌入式里的下一步，不应该只是继续把“写代码”这件事做得更快一点。

真正值得做的，是让它开始进入那些只有真实硬件现场才有的信息层：

- 板子有没有起来
- 哪一步初始化之后就没动静了
- 当前是在 wait loop 里，还是已经 fault
- probe 读到的寄存器和日志反映出来的现象是否一致
- 改完以后系统是不是真的恢复了

如果 AI 一直拿不到这些信息，它在嵌入式里就始终更像一个“会写 C 代码的助手”。

但如果它开始能读这些现场信息，它就会逐渐接近另一种角色：

**不是只会写代码，而是开始像一个调试工程师一样工作。**

## 现在它还很早，但已经不是空想

我不会把现在的结果吹成“AI 自动调试平台已经完成”。

现在的边界其实很明确：

- 只先验证了一块 `STM32L4`
- 主场景只先打通了一个 startup fault
- probe runtime 当前基于 `pyOCD`
- 当前实机验证使用的是 `ST-Link`
- 编译和烧录链路刚刚接进来
- 更复杂的电平、电压、波形、仪器能力还没接入

但即便这样，我也觉得这已经是一个真实的 MVP 了。

因为一个真实的 MVP，不需要一开始就又大又全。

它只需要先证明最关键的一件事：

**在一块真实板子上，这条链不是想象出来的，而是真的能跑。**

而这次我已经把它跑出来了：

**一句“板子灯不闪”，最后真的走到了排障、修复、编译、烧录和恢复验证。**

## 我接下来还会继续把这条链往前推

接下来我会继续沿着这条主线往下做，优先顺序大概是：

1. 把这次真实验证整理成更完整的 README 和 demo
2. 继续打磨 `startup failure / HardFault` 这条主路径
3. 把 `build / flash / reset / verify` 收成更稳定的闭环
4. 再逐步扩展到 GPIO、电平、电压、波形和仪器能力

如果这条路继续走下去，我希望最后形成的不是一个“能调某一块板子的小工具”，而是一层更通用的基础设施：

**AI 的嵌入式现场调试基础设施。**

## 最后

这次最让我有感觉的，不是“我又写了一些代码”，而是终于把一个我很想验证的判断，落在了一块真板上。

那就是：

**AI 在嵌入式里的下一步，不只是写代码，而是开始读板子、控工具、定位故障，并在修复后验证系统已经恢复。**

而 `mcudbg`，就是我往这个方向推的第一步。
