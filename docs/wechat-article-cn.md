# AI 写嵌入式代码还不够，我想让它开始直接调板子

这段时间，我一直在想一个问题：

为什么 AI 在嵌入式开发里，最容易被夸的能力，还是“会写代码”？

它当然能写代码。它可以写驱动、写状态机、写初始化、写外设配置，甚至能把很多模板化工作做得很快。

但真正做嵌入式的人都知道，最难的部分往往不是“把代码写出来”，而是“代码烧到板子上之后，到底发生了什么”。

这时候你关心的问题会立刻变成另一类东西：

- 板子有没有真的跑起来
- 日志停在了哪一步
- 探针能不能 attach
- 当前 PC 在哪
- 是卡死、跑飞，还是进了 HardFault
- fault 是无效指针、非法执行地址，还是外设访问异常

也就是说，一旦进入真实调试阶段，问题的核心就不再是“代码生成”，而是“板级观测与诊断”。

这也是我最近在做 `mcudbg` 的原因。

我想做的，不是一个让 AI 多会写一点嵌入式代码的工具，而是一个让 AI 开始真正参与“调板子”的工具。

## 我想解决的，不是代码问题，而是现场问题

我越来越觉得，AI 在嵌入式里真正缺的能力，不是继续提高几分代码质量，而是缺少对真实硬件现场的感知。

今天大部分 AI 工具仍然停留在这样的能力边界里：

- 会读代码
- 会改代码
- 会解释代码
- 会写测试

但它们通常还不会：

- 看一块真实板子的日志
- 连上真实 probe
- 读真实寄存器
- 判断当前故障发生在哪个阶段
- 根据 fault 寄存器和符号信息推断根因

所以我想做一个更贴近真实工程环境的东西。

`mcudbg` 的目标可以用一句话概括：

**给 AI 一层板级调试与观测能力。**

如果把这个目标讲得再直接一点，那就是：

**人负责接线，AI 负责读状态、控工具、定位问题。**

这件事如果做成，它就不再只是“AI 写嵌入式代码”，而是开始进入“AI 参与嵌入式现场调试”。

## 我没有一开始做大，而是先做一条很窄的闭环

这个方向的想象空间很大。

往大了说，它后面可以扩展到：

- UART / RTT / semihosting
- J-Link / ST-Link / CMSIS-DAP
- GPIO、电平检测
- 电压、电流观测
- 波形、逻辑分析仪、示波器
- 编译、烧录、复位、验证的自动闭环

但如果一开始什么都做，这个项目一定会散。

所以我把 `mcudbg v0.1` 刻意收得很窄，只做最有代表性的一条路径：

- 目标板：`STM32L496VETx`
- 探针：`ST-Link`
- probe runtime：`pyOCD`
- 日志通道：`UART`
- 符号文件：`ELF/AXF`
- 目标场景：启动阶段跑到 `sensor init...` 后进入 `HardFault`

我想先验证的，不是“这个愿景是不是很酷”，而是一个更现实的问题：

**AI/MCP 工具链能不能在一块真实板子上，看懂一次真实的 startup fault。**

## 我想证明的不是概念，而是真板闭环

为了做这个验证，我改了 STM32L4 的样板工程，让它构造出一个稳定、可重复的故障样本：

1. 板子正常打印启动日志
2. 程序执行到 `sensor init...`
3. 故意跳到非法执行目标
4. 进入 `HardFault`
5. 在 `HardFault_Handler` 里把现场通过 UART 打印出来

这样做的原因很简单。

如果板子侧没有一个真实、稳定、可复现的 fault 场景，那么后面所有 AI 诊断都可能只是“像在诊断”，但没有真实锚点。

而我想做的恰恰不是演示概念，而是证明这件事已经能落在真实硬件上。

## 真板实际打印出来的日志

这次板子实际打出来的日志如下：

```text
boot start
clock init ok
uart init ok
led init ok
sensor init...

[HardFault]
MSP  = 0x20000738
SP[0]= 0xFFFFFFFF
SP[1]= 0x08001975
SP[2]= 0x00010B61
SP[3]= 0x00010B61
SP[4]= 0x00013880
SP[5]= 0x000003E8
SP[6]= 0x00000001
SP[7]= 0x0800273D
CFSR = 0x00000001
HFSR = 0x40000000
MMFAR= 0xE000ED34
BFAR = 0xE000ED38
SHCSR= 0x00000000
```

对嵌入式开发者来说，这段输出已经非常有信息量了。

它说明了几件事：

- 板子确实跑到了 `sensor init...`
- 程序没有正常继续往下执行
- 板子确实进入了 `HardFault`
- `HardFault` 现场是可观测、可复盘的

换句话说，这不是“我猜它 fault 了”，而是板子自己把现场吐了出来。

## 然后我让 mcudbg 去读这个真实现场

在这块真板上，我实际跑通的 MCP 调用链是：

```text
list_demo_profiles()
load_demo_profile("stm32l4_atk_led_demo")
connect_with_config()
log_tail(30)
diagnose_startup_failure()
diagnose_hardfault()
```

这条链路里每一步都对应真实世界信号：

- `connect_with_config()` 连接 probe、UART 和 ELF
- `log_tail(30)` 读取真实启动日志和 fault 现场
- `diagnose_startup_failure()` 判断启动在哪一阶段失败
- `diagnose_hardfault()` 分析 fault 类型和根因方向

这次跑出来的结果，不只是“工具能连上”，而是真正给出了和板子状态一致的诊断。

核心结果包括：

- `startup_failure_with_fault`
- `hardfault_detected`
- fault class: `instruction_access_violation`
- `PC` 解析到 `HardFault_Handler`
- `LR` 解析到 `_printf_char_file`

这意味着，`mcudbg` 已经可以把：

- 真实 UART 日志
- 真实寄存器状态
- 真实 fault 寄存器
- 真实 ELF 符号信息

组合成一份对人有意义的诊断结果。

这一步对我来说非常关键。

因为它说明 `mcudbg` 已经不是停留在 mock 层、脚本层，或者 PPT 层，而是开始对真实板子产生价值。

## 这件事为什么值得做

我觉得，AI 在嵌入式里的下一步，不应该只是“把写代码这件事做得更快”。

真正值得做的，是让 AI 开始接触那些只有真实硬件现场才有的信息。

比如：

- 板子是否真的起来了
- 哪一步初始化后就没动静了
- 当前是卡在 wait loop 里，还是已经进入 fault
- fault 是代码路径问题，还是寄存器访问问题
- probe 看到的现场和日志里反映出来的现象是否一致

如果 AI 一直拿不到这些信息，它在嵌入式里就始终像一个“很会写 C 代码的助手”。

但如果它开始能读这些现场信息，它就会逐步接近另一种角色：

**不是只会写代码，而是开始像一个调试工程师一样工作。**

这也是我做 `mcudbg` 最兴奋的地方。

我不是想做一个“会调用某个探针命令的包装器”，而是想推进这样一件事：

**AI 能不能真正进入嵌入式板级调试这一步。**

## 现在它还很早，但已经不是空想

我不会把现在的结果吹成“AI 自动调试平台已经完成”。

现在它的边界其实非常明确：

- 只验证了一块 `STM32L4`
- 主场景只打通了一个 startup fault
- probe runtime 目前基于 `pyOCD`
- 真实验证当前使用的是 `ST-Link`
- `build / flash` 还没并进主线
- 电压、电平、波形、仪器这些能力还没接入

但即使这样，我也认为这已经是一个真实 MVP。

因为一个真正的 MVP，不需要一开始就又大又全。

它只需要证明最关键的一件事：

**在一块真实板子上，这条链路不是想象出来的，而是真的能跑。**

而现在，`mcudbg` 已经证明了：

**AI 可以通过 probe + log + ELF 上下文，对一块真实嵌入式板子的 startup fault 做出有意义的诊断。**

这就够了。

## 我接下来会继续做什么

接下来我会继续沿着这条主线往下推，优先顺序大概是：

1. 把这次真板验证整理成更完整的 README 和 demo
2. 继续打磨 `startup failure / HardFault` 这条主路径
3. 把 `build / flash / reset / verify` 并入工具链
4. 再逐步扩展到 GPIO、电平、电压、波形、仪器能力

如果这条线继续走下去，我希望最后形成的不是一个“能调某一块板子的小工具”，而是一层更通用的基础设施：

**AI 的嵌入式现场调试基础设施。**

## 最后

这次最让我有感触的，不是“我又写了一些代码”，而是终于把一个我很想验证的判断，落在了一块真板上。

那就是：

**AI 在嵌入式里的下一步，不只是写代码，而是开始读板子、控工具、定位故障。**

而 `mcudbg`，就是我想往这个方向推进的第一步。
