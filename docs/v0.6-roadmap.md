# v0.6 Roadmap

This document captures the current planned direction for `mcudbg` after the
core probe, ELF/DWARF, RTOS, RTT, flash, diagnosis, and GDB-server flows were
brought to a stable and hardware-validated state.

The goal of `v0.6` is not to rebuild the core.  
It is to deepen the strongest parts of the tool and reduce the remaining gaps in:

- J-Link advanced tracing
- multi-board robustness
- richer RTOS inspection
- AI-first workflows
- release quality documentation

---

## 1. Priority Summary

Recommended implementation order:

1. J-Link trace enhancements
2. device match / patch system
3. richer RTOS scenarios
4. stronger `diagnose(symptom)` workflows
5. AI examples and operator-facing examples
6. release-quality docs and demo materials

---

## 2. Theme A: J-Link Trace Enhancements

### Objective

Turn the J-Link backend from "fully usable for debug" into "the strongest backend for trace-oriented workflows".

### Current status

Already validated:

- connect / halt / reset / continue / step
- source-level debug
- flash erase / program / verify
- J-Link GDB server
- RTT
- DWT cycle counter

### Remaining goals

- `read_swo_log(...)`
- optional `read_itm_trace(...)`
- stronger stop-reason consistency
- trace-oriented examples and validation scripts

### Suggested sequence

1. `read_cycle_counter()` was completed first
2. add `read_swo_log(...)`
3. evaluate `ITM` support after SWO works on real hardware

### Definition of done

- unit tests
- real hardware validation
- `PROGRESS.md` updated
- `README.md` updated if the feature becomes externally useful

---

## 3. Theme B: Device Match / Patch System

### Objective

Reduce the amount of manual target guessing required by users and AI agents.

### Problems to solve

- target names are still too manual
- aliases are inconsistent across toolchains
- some boards need backend-specific connection workarounds
- reset / speed / serial selection can vary between environments

### Planned capabilities

- chip name alias table
- fuzzy target matching
- backend-specific connection fallback
- board-specific or chip-specific patch hooks

### Example outcomes

- `STM32F103C8T6` resolves cleanly to the expected backend target string
- common STM32 aliases no longer require trial-and-error
- problematic boards can auto-apply stable connect settings

---

## 4. Theme C: Richer RTOS Scenarios

### Objective

Extend RTOS validation beyond the currently verified queue / semaphore / timer-service flows.

### Already validated

- task listing
- task context reads
- queue receive
- semaphore wait
- software timer / `Tmr Svc`

### Remaining scenarios

- mutex
- event group
- task notification
- ISR-to-task notification

### Why this matters

These are common real-world blocked and wake-up paths, and they improve both:

- evidence quality for RTOS debugging
- realism of AI debugging workflows

---

## 5. Theme D: Stronger `diagnose(symptom)` Workflows

### Objective

Make the high-level diagnosis entrypoint feel more like a reliable debugging orchestrator.

### Current state

`diagnose(symptom)` already routes to useful lower-level tools.

### Remaining gaps

- better symptom taxonomy
- more stable multi-step diagnosis patterns
- better automatic combination of:
  - probe state
  - ELF/DWARF
  - SVD
  - RTT
  - RTOS

### End-state

Given a user report such as:

- "board won't boot"
- "UART is silent"
- "task is stuck"

the AI should be able to follow a clearer default tool sequence with less manual prompting.

---

## 6. Theme E: AI Examples

### Objective

Teach AI agents how to use `mcudbg` effectively through scenario-driven examples.

### Planned document

- `docs/ai-examples.md`

### Suggested initial scenarios

1. board will not boot
2. hardfault after reset
3. UART has no output
4. FreeRTOS task appears stuck
5. J-Link RTT validation

### Guideline

Each example should show:

- the user problem
- the recommended tool sequence
- the important returned fields
- how to interpret the evidence

---

## 7. Theme F: Release-Quality Documentation

### Objective

Make the project easier to adopt and evaluate from the outside.

### Remaining documentation gaps

- `README_CN.md` parity
- capability matrix
- backend support matrix
- board validation guide
- short demo / GIF / video

### Why this matters

At this stage, product clarity is becoming as important as raw feature count.

---

## 8. Milestone Proposal

### v0.6-alpha

Focus:

- J-Link trace start
- device match foundation

Expected outcome:

- J-Link gains at least one additional trace-facing tool beyond RTT and CYCCNT
- target selection becomes less manual

### v0.6-beta

Focus:

- richer RTOS scenarios
- stronger `diagnose(symptom)` orchestration

Expected outcome:

- broader real-world RTOS debugging coverage
- more reliable AI-directed diagnosis workflows

### v0.6

Focus:

- docs, examples, and release polish

Expected outcome:

- clearer external positioning
- easier onboarding for both humans and AI agents
