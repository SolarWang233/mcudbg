# mcudbg 30 / 60 / 90 天迭代路线图

这份路线图的目标，不是把功能列得越多越好，而是给 `mcudbg` 接下来 3 个月一个清楚的推进顺序：

- 先把已经跑通的真板闭环做稳
- 再扩故障场景
- 再扩硬件接入
- 同时持续输出内容，把技术进展转成影响力

核心原则只有一句话：

**每一轮迭代，都要同时产出“能力增长”和“对外证据”。**

也就是说，不只是写代码，还要留下：

- 可复现的 demo
- 可传播的视频 / GIF
- 可复用的文档
- 可讲清楚的项目进展

---

## 未来 30 天：把第一条闭环做稳

### 目标

把现在这条已经打通的链路，打磨成一个可以重复演示、可以反复验证、可以让别人快速理解的 MVP。

当前主线：

- 板子：`STM32L496VETx`
- probe：`ST-Link`
- probe runtime：`pyOCD`
- log：`UART`
- build / flash：`Keil UV4`
- 场景：`sensor init... -> HardFault -> 修复 -> 验证恢复`

### 这一阶段最重要的结果

1. `run_debug_loop` 变得更稳，能真正承担“一句现象 -> 一次诊断”的入口角色。
2. `build -> flash -> connect -> diagnose -> verify` 的顺序更稳定，不再容易被 probe/UART 占用问题打断。
3. 当前 demo 的录屏、GIF、README、文章形成一套对外展示材料。
4. 至少有一个“别人打开仓库就能看懂项目价值”的主页结构。

### 具体工作

- 打磨 `run_debug_loop`
  - 更好的阶段摘要
  - 更好的失败原因归纳
  - 更好的 cleanup 行为
- 增强生命周期管理
  - probe 占用释放
  - UART 占用释放
  - build/flash 前后的资源切换
- 优化诊断结果
  - 少一点原始字段堆砌
  - 多一点人类可读 summary
  - 明确下一步建议
- 补齐文档闭环
  - quickstart
  - demo 流程
  - 常见问题
  - README 首页叙事

### 这 30 天最该发的内容

- 1 篇真板首发文章
- 1 个完整闭环视频
- 2 张 GIF
- 2 到 3 条短动态

### 30 天阶段完成标准

满足下面这些条件，就说明第一阶段合格：

- 你自己可以稳定重复演示 3 次以上
- 外部读者看 README 能理解你在做什么
- 别人看到视频后能复述你的价值点
- `mcudbg` 不再像一堆零散 tool，而像一条完整 workflow

## 未来 60 天：扩场景，不急着扩太多硬件

### 目标

证明 `mcudbg` 不是只能处理一个精心构造的 demo fault，而是开始覆盖真实嵌入式开发里常见的启动期问题。

### 这一阶段的核心策略

**先扩故障类型，再扩设备类型。**

原因很简单：

- 扩 probe 并不会自动带来更强的产品价值
- 扩故障场景更容易证明“这不是只会跑一个演示脚本”

### 建议优先扩的故障场景

1. 启动卡死
   - 日志停在某一步
   - 没进入 HardFault
   - 主循环没跑起来

2. 上电无日志
   - 复位后没有任何串口输出
   - 需要区分“串口没通”和“系统没起来”

3. 外设初始化失败
   - 比如时钟、传感器、总线初始化失败
   - 日志提示明确，但需要结合寄存器或运行态判断

4. 死循环 / wait 卡住
   - 不是 fault
   - 但系统没有继续推进

### 这一阶段该补的高层能力

- `attempt_fix_and_verify`
  - 根据诊断结果尝试修复
  - 自动 build / flash / verify
- 更明确的问题分类
  - `startup_fault`
  - `startup_stall`
  - `no_boot_log`
  - `runtime_loop_stuck`
- 故障样本管理
  - 每个样本有自己的 trigger、预期日志、预期诊断

### 这 60 天最该发的内容

- “不仅能看 HardFault，还能看启动卡死”的第二波内容
- 不同故障类型的小视频 / 短 GIF
- 一篇更偏技术深度的文章
  - 比如：`AI 不是怎么写嵌入式代码，而是怎么读启动现场`

### 60 天阶段完成标准

- 至少有 3 个真实故障样本
- 每个样本都有稳定 demo
- `mcudbg` 不再只代表“AI 诊断 HardFault”
- 你开始有一套可复用的故障库

## 未来 90 天：扩硬件接入，开始往平台化走

### 目标

让 `mcudbg` 从“一个在 STM32L4 上很能打的 demo”开始往“更通用的 AI 板级调试基础设施”推进。

### 这一阶段建议优先扩的硬件能力

#### 1. probe 扩展

优先级最高：

- `J-Link`
- `CMSIS-DAP`
- 更稳定的 `ST-Link` 支持策略

因为 probe 扩展会直接影响：

- 可用板子范围
- 用户复现概率
- 传播时的说服力

#### 2. log 扩展

第二优先级：

- RTT
- semihosting
- 更通用的串口配置模型

#### 3. 板级观测扩展

第三优先级：

- GPIO / 电平
- 电压 / 电流
- 电源控制

#### 4. 仪器扩展

这一块放在后面：

- 示波器
- 逻辑分析仪
- 万用表

### 这一阶段该开始考虑的产品结构

- profile / board package 体系
- 统一配置模型
- 结果结构标准化
- 故障知识库沉淀
- 可扩展 backend 接口

### 这 90 天最该发的内容

- “从单板 demo 到多 probe 支持”的里程碑更新
- 一篇总结性质的长文
  - 讲清楚 `mcudbg` 为什么不是另一个 probe wrapper
- 一个更成熟的项目首页和对外演示页

### 90 天阶段完成标准

- 支持不止一种 probe 路径
- 支持不止一种故障类型
- 有更明确的平台化架构
- 外部已经开始把你和“AI 调板子”这个方向绑定起来

---

## 每个月都要重复的节奏

无论在 30、60 还是 90 天阶段，都建议保持同一套循环：

1. 做通一个新场景
2. 录一个 demo
3. 写一篇短内容
4. 收反馈
5. 根据反馈再补产品

这个节奏的好处是：

- 产品不是闭门造车
- 内容不是空喊口号
- 你每次都在积累下一次传播素材
- 项目会越来越有“在往前走”的感觉

---

## 你接下来最应该盯住的核心指标

不要一开始就盯 stars 或者转发数，先盯这几个更实在的指标：

- 同一个 demo 是否能稳定复现
- 一句自然语言是否能触发一条完整调试链
- 诊断结果是否越来越像人类工程师的结论
- 修复后验证是否越来越稳定
- 新样本接入成本是否越来越低

如果这几个指标在变好，项目就在往对的方向走。

---

## 一句话总结

`mcudbg` 后面的迭代，不应该是“再多加一些功能”，而应该是：

**先把第一条真实闭环做稳，再扩故障场景，再扩硬件能力，最后把它做成 AI 的嵌入式现场调试基础设施。**
